# 微信数据库存储格式详解

## 1. 整体架构

微信在本地使用 **SQLite 数据库**存储所有数据，但并非普通的 SQLite——数据库被微信**自研的加密层**包裹（WCDB，即 WeChat Database）。

### 存储位置

| 平台 | 数据目录 |
|------|---------|
| macOS | `~/Library/Containers/com.tencent.xinWeChat/Data/Documents/xwechat_files/<wxid>/db_storage/` |
| Windows | `%APPDATA%/Tencent/xwechat/config/*.ini` → `xwechat_files/<wxid>/db_storage/` |
| Linux | `~/Documents/xwechat_files/<wxid>/db_storage/` |

### 目录结构

```
xwechat_files/<wxid>/
├── db_storage/                    # 加密数据库目录
│   ├── contact/
│   │   ├── contact.db            # 联系人数据库
│   │   ├── contact.db-wal        # WAL 日志
│   │   └── contact.db-shm        # 共享内存
│   ├── session/
│   │   ├── session.db            # 会话数据库
│   │   └── session.db-wal
│   ├── message/
│   │   ├── message_0.db          # 消息数据库（可能有多个）
│   │   ├── message_0.db-wal
│   │   ├── message_1.db
│   │   └── message_1.db-wal
│   └── favorite/
│       ├── favorite.db           # 收藏数据库
│       └── favorite.db-wal
└── msg/                          # 媒体文件目录
    ├── attach/                   # 图片/语音/视频附件
    │   └── <hash>/
    │       └── YYYY-MM/
    │           ├── Img/
    │           ├── Voice/
    │           └── Video/
    ├── file/                     # 文件
    │   └── YYYY-MM/
    └── video/                    # 视频缩略图
        └── YYYY-MM/
```

## 2. 加密格式

每个 `.db` 文件不是标准 SQLite 头，而是微信自定义的加密格式。

### 加密参数

| 参数 | 值 |
|------|-----|
| 加密算法 | AES-256-CBC |
| 页面大小 | 4096 字节 |
| 密钥长度 | 32 字节 (256 位) |
| Salt 长度 | 16 字节 |
| Reserve 区域 | 80 字节 (IV 16 + HMAC-SHA512 64) |

### 单页结构（4096 字节）

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│  第1页特殊：前16字节为 Salt（解密后替换为SQLite头）     │
│  其余页：全部为加密数据                                │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │           加密数据区域（对齐到16字节）            │  │
│  │                                                │  │
│  │  第1页: [Salt 16B] [加密数据 ...]               │  │
│  │  其余页: [加密数据 ...]                         │  │
│  │                                                │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
│  ┌────────────────────────────────────────────────┐  │
│  │           Reserve 区域（末尾80字节）             │  │
│  │                                                │  │
│  │  [0..16]   → IV（初始化向量，16字节）           │  │
│  │  [16..80]  → HMAC-SHA512（完整性校验，64字节）  │  │
│  │                                                │  │
│  └────────────────────────────────────────────────┘  │
│                                                      │
└──────────────────────────────────────────────────────┘
```

### 密钥派生与验证

```
主密钥 (enc_key, 32字节)
    │
    ├── HMAC 密钥派生：
    │   mac_salt = salt XOR 0x3A
    │   mac_key = PBKDF2-HMAC-SHA512(enc_key, mac_salt, iterations=2)
    │
    └── 完整性验证：
        HMAC-SHA512(mac_key, page_data + page_number)
        → 与 Reserve 区域末尾 64 字节比对
```

### 第1页特殊处理

解密第1页后，需要将前16字节的 Salt 替换为标准 SQLite 文件头：

```
原始第1页:  [Salt 16B] [加密数据...] [IV 16B] [HMAC 64B]
解密后:     [SQLite Header 16B] [明文数据...] [Padding]
```

标准 SQLite 头标识：`SQLite format 3\x00`

## 3. WAL 日志加密

微信的 WAL (Write-Ahead Log) 文件同样被加密，结构与主库一致。

### WAL 文件结构

```
┌──────────────────────────────┐
│       WAL Header (32字节)     │
│  [0..4]   Magic Number       │
│  [4..8]   File Format        │
│  [8..12]  Page Size          │
│  [12..16] Checkpoint Seq     │
│  [16..20] Salt-1             │
│  [20..24] Salt-2             │
│  [24..28] Checksum-1         │
│  [28..32] Checksum-2         │
├──────────────────────────────┤
│       WAL Frame 1            │
│  ┌────────────────────────┐  │
│  │  Frame Header (24字节)  │  │
│  │  [0..4]   Page Number  │  │
│  │  [4..8]   Commit Size  │  │
│  │  [8..12]  Salt-1       │  │
│  │  [12..16] Salt-2       │  │
│  │  [16..20] Checksum-1   │  │
│  │  [20..24] Checksum-2   │  │
│  ├────────────────────────┤  │
│  │  加密页面 (4096字节)    │  │
│  │  (与主库页面格式相同)   │  │
│  └────────────────────────┘  │
├──────────────────────────────┤
│       WAL Frame 2            │
│       ...                    │
└──────────────────────────────┘
```

### WAL 解密流程

1. 读取 WAL Header，提取 `wal_salt1` 和 `wal_salt2`
2. 遍历每个 Frame：
   - 读取 Frame Header，获取 `pgno`（页号）和 `frame_salt1/2`
   - 验证 `frame_salt` 与 `wal_salt` 一致（防止读到过期数据）
   - 使用与主库相同的 `decrypt_page()` 解密页面数据
   - 将解密后的数据写回主库对应页号位置

## 4. 数据库文件清单

| 相对路径 | 用途 | 主要表 |
|---------|------|-------|
| `contact/contact.db` | 联系人信息 | `contact`, `chat_room`, `chatroom_member` |
| `session/session.db` | 会话列表 | `SessionTable` |
| `message/message_0.db` | 消息数据库（可能有多个，按编号递增） | `Msg_<md5hash>`, `Name2Id` |
| `message/message_1.db` | 消息数据库 | 同上 |
| `favorite/favorite.db` | 收藏 | `fav_db_item` |

## 5. 各数据库表结构

### 5.1 contact.db — 联系人

#### `contact` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 主键 |
| username | TEXT | 微信ID（如 `wxid_xxx`、`xxx@chatroom`、`gh_xxx`） |
| nick_name | TEXT | 昵称 |
| remark | TEXT | 备注名 |
| alias | TEXT | 微信号 |
| description | TEXT | 个人描述 |
| small_head_url | TEXT | 小头像URL |
| big_head_url | TEXT | 大头像URL |
| verify_flag | INTEGER | 认证标志 |
| local_type | INTEGER | 联系人类型 |

#### `chat_room` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | INTEGER | 关联 contact.id |
| owner | TEXT | 群主 username |

#### `chatroom_member` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| room_id | INTEGER | 关联 chat_room.id |
| member_id | INTEGER | 关联 contact.id |

### 5.2 session.db — 会话

#### `SessionTable` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| username | TEXT | 聊天对象ID |
| unread_count | INTEGER | 未读消息数 |
| summary | TEXT | 最后一条消息摘要 |
| last_timestamp | INTEGER | 最后消息时间戳（Unix） |
| last_msg_type | INTEGER | 最后消息类型 |
| last_msg_sender | TEXT | 最后消息发送者 username |
| last_sender_display_name | TEXT | 发送者显示名 |

### 5.3 message_*.db — 消息

#### `Msg_<md5hash>` 表

表名规则：`Msg_` + username 的 **MD5 哈希值**（32位十六进制）

例如：username 为 `wxid_abc123`，则表名为 `Msg_` + `md5("wxid_abc123")` 的十六进制表示

| 字段 | 类型 | 说明 |
|------|------|------|
| local_id | INTEGER | 本地消息ID |
| local_type | INTEGER | 消息类型（64位，低32位=基础类型，高32位=子类型） |
| create_time | INTEGER | 创建时间戳（Unix秒） |
| real_sender_id | INTEGER | 实际发送者ID（关联 Name2Id.rowid） |
| message_content | BLOB | 消息内容（可能被 zstd 压缩） |
| WCDB_CT_message_content | INTEGER | 内容压缩类型（4 = zstd 压缩） |

#### `Name2Id` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| rowid | INTEGER | 行ID（自增） |
| user_name | TEXT | 微信 username |

> 用途：`Msg_` 表中的 `real_sender_id` 关联此表的 `rowid`，用于解析群聊中每条消息的实际发送者。

### 5.4 favorite.db — 收藏

#### `fav_db_item` 表

| 字段 | 类型 | 说明 |
|------|------|------|
| local_id | INTEGER | 本地ID |
| type | INTEGER | 收藏类型 |
| update_time | INTEGER | 更新时间戳（Unix秒） |
| content | TEXT | XML 格式内容 |
| fromusr | TEXT | 来源用户 username |
| realchatname | TEXT | 来源聊天 username |

收藏类型编码：

| type | 含义 |
|------|------|
| 1 | 文本 |
| 2 | 图片 |
| 5 | 文章/链接 |
| 19 | 名片 |
| 20 | 视频号 |

## 6. 消息类型编码

`local_type` 是 64 位整数，按位拆分：

```
┌─────────────────────────────────┬─────────────────────────────────┐
│       高32位：子类型              │       低32位：基础消息类型        │
│     (local_type >> 32)          │    (local_type & 0xFFFFFFFF)    │
└─────────────────────────────────┴─────────────────────────────────┘
```

### 基础消息类型（低32位）

| 类型值 | 含义 |
|-------|------|
| 1 | 文本 |
| 3 | 图片 |
| 34 | 语音 |
| 42 | 名片 |
| 43 | 视频 |
| 47 | 表情/贴纸 |
| 48 | 位置 |
| 49 | 链接/文件/小程序/引用（由子类型区分） |
| 50 | 通话 |
| 10000 | 系统消息 |
| 10002 | 撤回消息 |

### 子类型（高32位，仅 type=49 时有意义）

| 子类型 | 含义 |
|-------|------|
| 5 | 链接 |
| 6 | 文件 |
| 33 | 小程序 |
| 36 | 小程序 |
| 44 | 小程序 |
| 57 | 引用消息 |

## 7. 消息内容格式

### 7.1 压缩

`message_content` 字段可能被压缩：

| `WCDB_CT_message_content` 值 | 含义 | 处理方式 |
|-----|------|---------|
| 4 | zstd 压缩 | 使用 zstd 解码后得到 UTF-8 文本 |
| NULL 或其他 | 未压缩 | 直接读取为 UTF-8 文本 |

### 7.2 群聊消息格式

群聊消息的 `message_content` 以 `sender:\n` 前缀标识发送者：

```
wxid_sender123:\n这是消息的实际内容
```

解析规则：以 `:\n` 分割，第一部分为发送者 username，第二部分为消息正文。

### 7.3 XML 内容

链接(type 49)、通话(type 50)等内容为 XML 格式：

**链接/文件消息示例**：
```xml
<appmsg sdkver="0" appid="" ...>
  <title>文章标题</title>
  <des>文章描述</des>
  <type>5</type>
  <url>https://...</url>
  ...
</appmsg>
```

**引用消息示例**：
```xml
<appmsg ...>
  <title>引用者的回复</title>
  <type>57</type>
  <refermsg>
    <displayname>被引用者昵称</displayname>
    <content>被引用的原始消息</content>
  </refermsg>
</appmsg>
```

**通话消息示例**：
```xml
<voip ...>
  <duration>120</duration>
  ...
  Canceled / Line busy / not answered
</voip>
```

### 7.4 媒体路径解析

媒体文件存储在 `db_storage` 的同级 `msg/` 目录下，按 **年月** 分子目录：

| 消息类型 | 存储路径模式 |
|---------|------------|
| 图片 | `msg/attach/<hash>/YYYY-MM/Img/` |
| 语音 | `msg/attach/<hash>/YYYY-MM/Voice/` |
| 视频 | `msg/attach/<hash>/YYYY-MM/Video/` |
| 视频缩略图 | `msg/video/YYYY-MM/*_thumb.jpg` |
| 文件 | `msg/file/YYYY-MM/<filename>` |

> 文件名通过解析 XML 内容中的 `<title>` 标签获取。

## 8. 用户名格式

微信的 username 有多种格式，可用于区分联系人类型：

| 格式 | 类型 | 示例 |
|------|------|------|
| `wxid_<hash>` | 个人好友 | `wxid_abc123def456` |
| `xxx@chatroom` | 群聊 | `12345678@chatroom` |
| `gh_<hash>` | 公众号 | `gh_abc123def456` |
| `xxx@openim` | 开放平台账号 | - |

判断规则：
- 包含 `@chatroom` → 群聊
- 以 `gh_` 开头 → 公众号
- 以 `wxid_` 开头 → 个人好友

## 9. 密钥提取

### 9.1 提取原理

微信运行时将数据库密钥保存在进程内存中。密钥提取通过以下步骤实现：

1. **扫描微信进程内存**，匹配连续的十六进制字符串模式
2. **模式匹配**：
   - 96 位十六进制：前 64 位为 `enc_key`，后 32 位为 `salt`
   - 64 位十六进制：尝试作为 `enc_key`，与所有未匹配 salt 逐个验证
   - 更长字符串：截取前 64 位和末尾 32 位
3. **验证**：使用 PBKDF2 派生 HMAC 密钥，计算 HMAC-SHA512，与数据库中存储的 HMAC 比对
4. **交叉验证**：已找到的密钥尝试匹配其他 salt（不同数据库可能共享同一密钥）

### 9.2 macOS 实现

macOS 使用 C 二进制 `find_all_keys_macos` 提取密钥：

```
1. 扫描 db_storage/ 目录下所有 .db 文件
2. 读取每个文件第1页，提取前16字节作为 Salt
3. 构建 Salt → DB文件 映射
4. 调用 C 二进制扫描 WeChat 进程内存
5. C 二进制输出 all_keys.json（包含每个 DB 的 enc_key 和 salt）
6. 交叉验证未匹配的 salt
```

### 9.3 密钥文件格式

`all_keys.json` 格式：

```json
{
  "contact/contact.db": {
    "enc_key": "abc123...（64位十六进制）",
    "salt": "def456...（32位十六进制）",
    "size_mb": "12.3"
  },
  "message/message_0.db": {
    "enc_key": "...",
    "salt": "...",
    "size_mb": "256.7"
  }
}
```

## 10. 解密完整流程

```
原始加密 .db 文件
    │
    ▼
┌─────────────────────────────────────┐
│ 1. 读取密钥文件，获取 enc_key        │
│ 2. hex 解码 enc_key 为 32 字节密钥   │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 3. 逐页读取（4096 字节/页）          │
│ 4. 提取末尾 Reserve 区域的 IV       │
│    (page[4016..4032])               │
│ 5. AES-256-CBC 解密                 │
│    - 第1页：跳过前16字节Salt         │
│    - 其余页：全部数据               │
│ 6. 第1页：替换 Salt 为 SQLite 头    │
└─────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────┐
│ 7. 处理 WAL 日志（如存在）           │
│    - 读取 WAL Header (32字节)        │
│    - 遍历 WAL Frame                 │
│    - 验证 salt 一致性               │
│    - 解密每个页面                    │
│    - 写回主库对应位置               │
└─────────────────────────────────────┘
    │
    ▼
标准 SQLite 文件（可用 rusqlite/sqlite3 正常查询）
    │
    ▼
┌─────────────────────────────────────┐
│ 8. 缓存管理（DBCache）              │
│    - 以 (db_mtime, wal_mtime) 检测  │
│    - 文件未变化时复用缓存            │
│    - 缓存存储在系统临时目录          │
│    - 持久化 mtime 信息到 JSON        │
└─────────────────────────────────────┘
```

## 11. 缓存机制

为避免重复解密，系统实现了 `DBCache` 缓存机制：

| 特性 | 说明 |
|------|------|
| 缓存位置 | 系统临时目录 `/tmp/wechat_reader_cache/` |
| 缓存文件名 | `MD5(相对路径)[:12].db` |
| 失效检测 | 比较源文件和 WAL 文件的 mtime |
| 持久化 | `_mtimes.json` 记录所有缓存条目的 mtime |
| 跨会话复用 | 应用重启后加载持久化信息，跳过未变化的文件 |

缓存判断逻辑：

```
if 缓存存在 && db_mtime 未变 && wal_mtime 未变:
    直接使用缓存的解密文件
else:
    重新解密并更新缓存
```
